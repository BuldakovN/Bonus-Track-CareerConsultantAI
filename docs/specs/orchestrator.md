# Спецификация модуля: Оркестратор (`core`)

## Назначение

Оркестратор управляет полным жизненным циклом пользовательского запроса: валидацией, маршрутизацией, вызовами retrieval/tools/LLM, проверками качества и сохранением состояния.

## Ответственность модуля

- Нормализация входного запроса.
- Загрузка и обновление состояния сессии через `crud-service`.
- Выбор ветки исполнения по `intent` и текущему `user_state`.
- Координация `dialog-graph`, retrieval-контура и tool-вызовов.
- Применение guardrails и fallback-механик.
- Возврат финального ответа в транспортный слой.

## Входы и выходы

- Вход:
  - `user_id`, `prompt`, `parameters`.
- Выход:
  - `msg`, `professions`, `test_info`, `meta` (опционально: режим деградации, причина fallback).

## Контракты интеграции

- С `tg-module`: HTTP API `core` (`/v1/dialog/turn`, `/v1/dialog/clean`, совместимые legacy-пути).
- С `crud-service`: чтение/запись session snapshot.
- С `llm-service`: вызов генерации (`chat`) и планирования tool-call (при необходимости).
- С retrieval/tool-слоем: внутренние адаптеры с типизированными контрактами.

## Правила переходов и остановки

- Переходы определяются связкой `intent + user_state + guard_result`.
- Остановка исполнения:
  - успешный валидированный ответ;
  - безопасный отказ по политике;
  - деградированный ответ после исчерпания retry budget.

## Ошибки и fallback

- Ошибка LLM: 1 повтор -> fallback-профиль модели -> безопасный деградированный ответ.
- Ошибка retrieval/tools: локальный fallback ветки без полного прерывания turn.
- Ошибка CRUD: stateless-ответ и запрет рискованных изменений состояния.

## Контроль качества

- CP1: входная валидация.
- CP2: входной guard.
- CP3: контроль качества retrieval.
- CP4: контроль безопасности tools.
- CP5: валидация выхода модели.
- CP6: атомарность сохранения.
- CP7: полнота телеметрии.
