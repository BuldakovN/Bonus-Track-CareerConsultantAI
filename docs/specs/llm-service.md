# Спецификация модуля: LLM-сервис (`llm-service`)

## Назначение

`llm-service` предоставляет унифицированный API для инференса модели и (опционально) планирования tool-вызовов без владения бизнес-оркестрацией.

## Ответственность модуля

- Прием структурированных запросов на генерацию.
- Вызов провайдера модели с заданным профилем.
- Возврат результата генерации и usage-метрик.
- Нормализация ошибок провайдера.

## Что вне ответственности

- Управление state machine диалога.
- Принятие решений о маршрутизации, fallback-ветках и persist.
- Работа с БД и транспортным слоем Telegram.

## API-контракты (PoC)

- `POST /v1/chat`
  - вход: список сообщений (`role`, `text`) + параметры генерации;
  - выход: `text`, опционально `token_usage`.
- `POST /v1/tool_call` (опционально)
  - вход: сообщение + профиль доступных tools;
  - выход: структурированный план tool-вызова.

## Ограничения и SLO

- Таймаут вызова модели по умолчанию: 8s.
- Максимум 1 повтор в вызывающем сервисе (`core`), не внутри `llm-service`.
- p95 по инференсу должен соответствовать глобальному бюджету turn в `core`.

## Обработка ошибок

- Нормализованные категории: timeout, provider_error, validation_error, unavailable.
- Ошибки возвращаются без утечки внутренних секретов/stack trace.
- При 5xx/timeout инициатор (`core`) применяет fallback-политику.

## Наблюдаемость

- Логи по запросу: trace/request id, latency, модель, token usage, категория ошибки.
- Метрики: доля успеха, p50/p95 задержки, частота таймаутов и ошибок провайдера.
