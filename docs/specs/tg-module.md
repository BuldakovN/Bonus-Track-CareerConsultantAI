# Спецификация модуля: TG-модуль (`tg-module`)

## Назначение

`tg-module` является транспортным и интерфейсным слоем: принимает события Telegram, формирует UX-ответы и проксирует доменную обработку в `core`.

## Ответственность модуля

- Прием входящих сообщений/команд/callback-кнопок Telegram.
- Вызов API `core` с нормализованным payload.
- Отображение ответов пользователю (текст, кнопки, подсказки).
- Локальная обработка ошибок транспорта (таймаут Telegram API, повтор отправки).

## Что модуль не делает

- Не содержит бизнес-логики маршрутизации диалога.
- Не принимает архитектурные решения по памяти/состоянию.
- Не вызывает `llm-service` и БД напрямую.

## Контракты

- Вход: Telegram Update.
- Выход:
  - HTTP-запрос в `core` (`/v1/dialog/turn` и смежные endpoint-ы);
  - Telegram message/callback response пользователю.

## Надежность и ограничения

- Ретрай отправки в Telegram ограничен (чтобы не создавать дубликаты).
- Пользователь получает дружелюбный fallback-текст при недоступности `core`.
- Логи модуля не содержат секретов и персональных данных в открытом виде.

## Точки контроля

- Корректная сериализация `parameters` для `core`.
- Идемпотентная обработка повторных update-id.
- Соответствие форматирования Telegram-ограничениям (длина текста, markup).
