# Спецификация: Serving / Config

## Назначение

Зафиксировать runtime-топологию, контракты конфигурации, работу с секретами и дисциплину версионирования моделей.

## Runtime-сервисы

- `core` (API оркестратора)
- `llm-service` (API инференса модели)
- `crud-service` (хранения состояний)

## Модель конфигурации

Источники конфигурации:
- переменные окружения;
- версионируемый `env.example` как базовая схема;
- runtime-секреты из `.env` или secret manager.

Обязательные категории:
- endpoints (`CRUD_BASE_URL`, `LLM_BASE_URL`, `WEB_SEARCH_API_URL`);
- timeouts/retries;
- профиль модели (`model_id`, temperature, max tokens);
- параметры retrieval (`top_k`, thresholds);
- feature flags (включение tools, fallback-модели, строгой валидации выхода).

## Политика секретов

- Секреты хранятся только в environment/secret manager, не в репозитории.
- Значения секретов не должны попадать в логи, трейсы, prompts и тексты исключений.

## Версионирование

- Профиль модели должен быть версионирован (`MODEL_PROFILE_VERSION`).
- Prompt/policy-шаблоны должны иметь тег семантической ревизии.
- API-контракты должны быть обратно совместимы для PoC-клиентов или публиковаться под версионированным путем.

## Операционные ограничения

- Целевые p95 по задержке:
  - стандартный turn <= 4s;
  - retrieval turn <= 6s.
- Целевая доступность: >= 99.0% успешных ответов.
- Контроль стоимости: ограничение токенов на turn + fallback в concise-режим.
