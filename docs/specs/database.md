# Спецификация модуля: База данных (`crud-service` + SQLite)

## Назначение

Модуль обеспечивает персистентное хранение состояния сессии, истории диалога и связанных пользовательских метаданных.

## Архитектура модуля

- Внешний интерфейс: `crud-service` (HTTP API).
- Физическое хранение: SQLite (volume в Docker).
- Модель доступа: только через CRUD API, прямой доступ других сервисов к SQLite запрещен.

## Хранимые данные

- Идентичность пользователя (`provider`, `external_user_id`, `app_user_id`).
- Сессионное состояние (`user_state`, `user_type`, metadata).
- История диалога (ограниченная по объему, с метками времени).
- Производные данные (результаты теста, рекомендации).

## API-контракты (минимум)

- `GET/PUT /v1/identities/{provider}/{external_user_id}/session`
- `DELETE /v1/identities/{provider}/{external_user_id}`
- Legacy-совместимость: `GET/PUT /users/{user_id}/session`

## Ограничения

- SQLite ограничивает параллелизм записи.
- Для PoC допускается single-node persistence без репликации.
- Время ответа CRUD должно быть <= 1.0s p95 на типовой нагрузке.

## Надежность и целостность

- Обновление сессии выполняется атомарно на уровне CRUD-операции.
- При недоступности БД `core` переходит в stateless-ответ (без рискованных мутаций).
- Схема БД версионируется, миграции обратимо документируются.

## Безопасность

- Нет хранения секретов в пользовательских полях.
- Логи CRUD редактируют PII и не пишут полный текст приватных сообщений.
