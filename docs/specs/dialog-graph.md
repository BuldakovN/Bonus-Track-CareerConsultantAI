# Спецификация модуля: Диалоговый граф (`core` / LangGraph)

## Назначение

Диалоговый граф реализует пошаговый сценарий обработки turn с явными узлами, ветвлениями и переходами между фазами пользовательского диалога.

## Основные узлы

- `prepare`: подготовка контекста, загрузка сессии, нормализация входа.
- `guard`: проверка prompt-injection/jailbreak и policy-ограничений.
- `route`: выбор ветки (`dialog`, `clean`, `profession`, `roadmap`).
- `retrieve` (условно): поиск, rerank, упаковка контекста.
- `llm_call`: формирование prompt и вызов `llm-service`.
- `validate_output`: структурная и safety-проверка ответа.
- `persist`: запись состояния в CRUD.
- `respond`: возврат результата вызывающему API.

## Состояния и переходы

- Бизнес-фазы: `who`, `about`, `test`, `recommendation`, `talk`, `inject_attempt`.
- Переходы зависят от:
  - текущей фазы;
  - типа входа (`intent`);
  - результата guard/валидации.

## Контракты узлов

- Каждый узел получает общий объект состояния графа (`GraphState`) и возвращает обновленный `GraphState`.
- Узлы не должны иметь скрытых побочных эффектов вне явных операций `persist`.
- В узлах, вызывающих внешние сервисы, обязательны timeout/retry и нормализация ошибок.

## Ошибки и деградации

- Сбой в `retrieve`: переход в ответ с пониженной уверенностью.
- Сбой в `llm_call`: повтор с сокращенным контекстом.
- Сбой в `validate_output`: повтор генерации или безопасный fallback.
- Сбой в `persist`: ответ без фиксации состояния + критический telemetry-флаг.

## Критерии качества графа

- Переходы полностью детерминированы правилами маршрутизации.
- Нет недостижимых веток и циклов без stop-condition.
- Каждая ветка завершается либо `respond`, либо безопасным отказом.
