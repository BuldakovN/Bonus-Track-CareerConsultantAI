# Governance: риски, логи, персональные данные, защита

## 1. Risk register

## 2. Политика логов и работа с персональными данными

### Логирование

- **Что логировать:** события (старт/стоп бота, команды пользователя, факт вызова API, ошибки), идентификатор пользователя (`user_id`), длительность запросов к LLM, счётчики токенов. Для отладки допустимо краткое упоминание типа запроса (например, «запрос описания профессии»), без полного текста сообщений и без сохранения полного контента диалога в логах.
- **Чего избегать в логах:** полный текст сообщений пользователя и ответов ассистента, имя/возраст и иные персональные данные в открытом виде. При необходимости отладки — только в защищённой среде с ограниченным доступом и сроками хранения.
- **Хранение:** логи приложения (например, `./logs`) не должны содержать персональные данные в объёме, достаточном для идентификации личности; срок хранения логов задаётся операционно (например, 30–90 дней) с последующим удалением или обезличиванием.
- **Мониторинг:** Prometheus собирает метрики (запросы, задержки, токены) без привязки к содержанию сообщений; Grafana-дашборды не должны отображать персональные данные.

### Персональные данные

- **Какие данные обрабатываются:** идентификатор пользователя Telegram (`user_id`), история сообщений диалога, метаданные (собранный портрет: тип, имя, возраст, интересы, результаты тестов, рекомендации профессий). Данные хранятся в SQLite (таблицы `conversation_history`, `user_metadata`).
- **Цель обработки:** обеспечение работы профориентационного диалога и персональных рекомендаций.
- **Минимизация:** не собирать избыточные персональные данные; в логах — только идентификаторы и метаданные, необходимые для эксплуатации и расследования инцидентов.
- **Права пользователя:** команда `/clean_history` обеспечивает удаление истории и метаданных пользователя для данного `user_id` (очистка в репозитории и в памяти Model). При масштабировании необходимо документировать процедуру удаления по запросу и сроки хранения.
- **Передача третьим лицам:** содержимое диалога и портрет передаются провайдерам LLM (Yandex/OpenAI/Anthropic/Google) в рамках формирования запросов к API; веб-поиск может получать запросы, сформированные на основе контекста. Необходимо учитывать политики этих провайдеров и при необходимости информировать пользователя.

## 3. Защиты от injection, jailbreak, утечки системного промпта (граф Chatsky) и подтверждение действий

### Роль диалогового графа Chatsky

Управление состояниями агента вынесено в **диалоговый граф Chatsky** (узлы: WHO, ABOUT, TEST, RECOMMENDATION, TALK). Это даёт единый механизм защиты:

- **Prompt injection:** пользовательский ввод никогда не подставляется в системный промпт. Системный промпт задаётся только узлом графа и передаётся в LLM как отдельное system-сообщение; пользовательское сообщение идёт отдельно (human). Переход в следующий узел выполняется по **условиям графа и результатам tool calls** (например, `userType`, флаг [EXIT]), а не по свободному тексту ответа LLM, поэтому попытки «инструктировать» модель сменить роль или выдать системный промпт не меняют узел и не подмешиваются в следующую системную инструкцию.
- **Jailbreak:** граф жёстко определяет, какой узел активен и какой промпт применяется. Пользователь не может через ввод переключить агента в произвольный режим: следующий узел выбирается кодом по условиям перехода (и по валидированным результатам tools), а не по содержанию ответа модели.
- **Утечка системного промпта:** системный промпт не конкатенируется с пользовательским вводом в одном сообщении; он привязан к узлу и передаётся из конфига/кода. Модель не получает единый блок «инструкция + пользователь», из которого могла бы «процитировать» системную часть в ответе.

### Дополнительные меры (код и конфиг)

- **Системные промпты узлов:** чётко задают роль и правила (не выдумывать за пользователя, не выполнять инструкции из пользовательского ввода). Хранятся в конфиге графа/узлов, не формируются из пользовательского ввода.
- **Структура вызова LLM:** история собирается в коде; в LLM уходят отдельно system-/human-/assistant-сообщения. Не подставлять сырой ввод пользователя в начало или в середину системного промпта.
- **Tool calls:** результаты выполнения тулов (например, тип пользователя, список профессий) интерпретируются программой; не выполнять произвольный код или внешние запросы на основе непроверенного вывода LLM. Валидация и маппинг enum-значений (school/student/worker, названия тестов) — на стороне приложения; переход по графу — только по этим валидированным результатам.
- **RAG:** запрос к индексу формируется из контролируемых данных (название профессии из рекомендаций, текст из индекса); при использовании веб-поиска — ограничение формата запроса и длины.

### Санитизация вывода для пользователя

- В **Telegram-боте** перед отправкой ответа LLM обрабатывается методом `sanitize_text`: удаление/замена символов разметки и ссылок, чтобы снизить риск отображения нежелательного контента и разметки от модели (в т.ч. при попытках вывести фрагменты системного промпта через ответ). Длинные сообщения разбиваются на части по длине.

### Подтверждение деструктивных и необратимых действий

- **Очистка истории (`/clean_history`):** удаляет всю историю диалога и метаданные пользователя. Сейчас выполняется по одному нажатию команды без дополнительного подтверждения. **Рекомендация:** перед вызовом API очистки показывать пользователю подтверждение («Вы уверены, что хотите удалить всю историю и начать заново? Да / Нет») и вызывать API только после явного «Да». Это снижает риск случайной потери данных и повышает осознанность действия.
- **Отмена операции (`/cancel`):** сбрасывает активность сессии (например, процесс тестирования); данные в БД не удаляются. Подтверждение не обязательно, но можно добавить короткое сообщение «Текущее действие отменено».

При внедрении подтверждения для `/clean_history` остаточный риск по «несанкционированной очистке» и случайному удалению снижается.
